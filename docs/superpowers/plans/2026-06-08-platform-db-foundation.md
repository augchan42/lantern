# Platform-DB Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **⚠️ Live-DB steps are marked `[YOU RUN — LIVE DB]`.** They operate on the shared
> production project `ezlyfsgpcahlnbqgdlxh` (hosts live photocritic + ara) and require the
> operator's Supabase auth + DB password. An agent must STOP at these and hand off — do not
> run them autonomously. Every such step has a verification gate before anything
> irreversible.

**Goal:** Turn the already-shared Supabase project `ezlyfsgpcahlnbqgdlxh` into a canonical
platform: one source-of-truth migration repo (`platform-db`), a captured baseline of the
current live schema, and the `ai_provider_requests` columns fix (promote `session_id` /
cost / tokens to top-level), without disrupting live apps.

**Architecture:** Adopt the existing project **in place** (no data move, no cutover). A new
`platform-db` repo owns `supabase/migrations/`. `supabase db pull` captures current live
schema as a baseline; legacy migration folders in photocritic-site/ara-eval are frozen. All
forward migrations (starting with the columns fix) live in `platform-db` and are additive
/ backward-compatible so photocritic + ara keep working untouched.

**Tech Stack:** Supabase CLI 2.75.0, Postgres, TypeScript (reference AI service ported from
`photocritic-site/src/lib/ai-provider.ts` + `fal-image.ts`).

**Scope:** Platform foundation + columns fix only. The `lantern_*` tables and the Lantern
app are a **separate follow-on plan** (`platform-db` will already be the place they land).

-- Promote session/cost/token values (previously buried in metadata jsonb) to
-- first-class, queryable columns on the shared AI request log. Additive + nullable,

```ts
export async function logAIProviderRequest({
  request_id,
  provider,
  model,
  model_array,
  use_case,
  response_status,
  error_message,
  raw_request,
  raw_response,
  metadata,
  session_id = null,
  total_cost = null,
  input_tokens = null,
  output_tokens = null,
  total_tokens = null,
}: {
  request_id: string;
  provider: string;
  model: string | null;
  model_array: string[] | null;
  use_case: string;
  response_status: number | null;
  error_message: string | null;
  raw_request: unknown;
  raw_response: unknown;
  metadata?: unknown;
  session_id?: string | null;
  total_cost?: number | null;
  input_tokens?: number | null;
  output_tokens?: number | null;
  total_tokens?: number | null;
}) {
  try {
    const supabase = createServerClient();
    const { error } = await supabase.from("ai_provider_requests").insert({
      request_id,
      provider,
      model,
      model_array,
      use_case,
      response_status,
      error_message,
      raw_request,
      raw_response,
      metadata,
      session_id,
      total_cost,
      input_tokens,
      output_tokens,
      total_tokens,
    });
    if (error) {
      console.error(`[AIProvider][${request_id}] Failed to log to DB:`, error);
    }
  } catch (err) {
    console.error(`[AIProvider] Logging error:`, err);
  }
}
```

Then, at the **success** call site (currently passing `usage` inside `metadata`), also pass
the top-level token fields so new apps populate them. Add these keys to that

        session_id: undefined,                       // apps pass their session id here
        input_tokens: result.usage?.input_tokens ?? null,
        output_tokens: result.usage?.output_tokens ?? null,
        total_tokens: result.usage?.total_tokens ?? null,
        // total_cost: populated when the provider returns cost (e.g. OpenRouter

- [ ] **Step 3: Write the logging contract doc**

Create `reference/ai/ai_provider_requests.contract.md`:
```markdown
# ai_provider_requests — logging contract

Shared table. Every AI call (success AND failure) inserts one row.

**Top-level columns (query these directly):**
- `request_id` text, unique — `crypto.randomUUID()`; failed attempts use
  `${request_id}_attempt_${n}`.
- `use_case` text — `<app>_<action>` (e.g. `lantern_scene`, `photo_critique`).
- `session_id` text — the app's session/grouping id (e.g. `lantern_sessions.id`).
- `model`, `model_array`, `provider`, `response_status`, `error_message`.
- `input_tokens`, `output_tokens`, `total_tokens` int — from provider usage.
- `total_cost` numeric — provider cost when available, else null.

**`metadata` jsonb:** anything app-specific (attempt_number, timings, raw usage blob…).
**`raw_request` / `raw_response`:** trimmed request + full provider response.
```

# Onboarding a new app to the platform

1. **Env:** point the app at the shared project —
   `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,

   `<app>AiService`; log with `use_case = <app>_<action>` and the app's `session_id`
   (see `reference/ai/ai_provider_requests.contract.md`).
4. **RLS:** enable RLS on `<app>_*`; writes go through the service-role admin client
   (single-operator pattern); no public client access.
