1. File path: `README.md:8-18`, `AGENTS.md:6-18`, `.claude/review-loop.local.md:8`, `docs/superpowers/plans/2026-06-08-lantern-foundation.md:5-35`, `package.json:8-21`
   Severity: `high`
   Category: `Holistic`
   Description: Repo guidance is materially out of sync with the actual codebase. The top-level docs still say the app is not scaffolded, the active review-loop instruction still points agents at the foundation plan as if the scaffold work is pending, and the foundation plan still starts with `create-next-app` instructions and claims `Next.js 15`, while the repository already contains a working `src/` app on `next@16.2.7`. 2. File path: `src/lib/supabaseAdmin.ts:1`, `src/services/openrouter.ts:1`, `src/services/falImage.ts:1`, `src/services/aiProviderLog.ts:1`
   Severity: `high`
   Category: `Next.js`
   Description: Secret-bearing server modules are not marked `server-only`. In an App Router project, that leaves the service-role Supabase client and provider integrations available for accidental client import, weakening server/client boundaries and increasing the chance of build-time or runtime exposure mistakes around privileged code.
   Suggested fix: Add `import "server-only";` to each server-only module that depends on service-role credentials or server-side providers, and keep client-safe wrappers separated from these modules.

### Medium

3. File path: `src/`, `docs/superpowers/`
   Severity: `medium`
   Category: `Holistic`
   Description: The repo now has two major working trees with different responsibilities, but neither has directory-local `AGENTS.md` / `CLAUDE.md` guidance. As the codebase grows, agents entering `src/` or `docs/superpowers/` directly will miss local conventions around server-only code, env handling, tests, and document ownership.
   Suggested fix: Add concise `AGENTS.md` files under `src/` and `docs/superpowers/`, and add `CLAUDE.md` symlinks for compatibility.

4. File path: `src/services/falImage.ts:65-70`, `src/services/falImage.ts:123-132`
   Severity: `medium`
   Category: `Diff, Next.js`
   Description: `fetchImageBuffer()` fetches and uploads whatever URL the provider returns without validating the scheme, host, response status, or MIME type. That leaves a server-side fetch sink that can persist non-image payloads to Blob storage and broadens the trust boundary around provider-returned URLs.
   Suggested fix: Require `https:`, allowlist expected FAL image hosts, reject non-2xx responses, and verify an `image/*` content type before converting the body to a buffer and uploading it.

5. File path: `src/services/aiProviderLog.ts:14-18`, `src/lib/lanternEvents.ts:5-21`
   Severity: `medium`
   Category: `Diff`
   Description: Both logging/event helpers accept raw string session IDs and write them directly to DB-backed tables while swallowing insert failures with `console.error`. If callers pass malformed IDs or inserts fail, the primary action appears to succeed but observability data is silently lost.
   Suggested fix: Validate or coerce `sessionId`/`session_id` at the service boundary, make the expected identifier format explicit in types, and emit a metric or propagate a structured failure signal instead of only logging to stderr.

6. File path: `src/services/openrouter.ts:110-192`, `src/services/falImage.ts:73-204`
   Severity: `medium`
   Category: `Diff`
   Description: The new AI integration paths do not have direct tests for their network/storage/logging behavior or failure branches. The existing suite covers routing and pure helpers, but not model fallback, timeout handling, blob upload behavior, or logging failures, so regressions in the most failure-prone paths can pass unnoticed.
   Suggested fix: Add mocked tests around `fetch`, Supabase inserts, and Vercel Blob uploads that cover success, timeout, provider error, invalid response, and logging-failure paths.

7. File path: `src/app/layout.tsx:15-18`
   Severity: `medium`
   Category: `Next.js`
   Description: The root metadata still uses the default `create-next-app` title and description. That means every route currently inherits placeholder SEO and browser-tab text unrelated to the Lantern product.
   Suggested fix: Replace the default metadata with Lantern-specific values now, or move to `generateMetadata()` once route-specific metadata is needed.
